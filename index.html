Introduction

 Dans ce cours, nous allons parler de système de gestion de version de manière générale, et plus précisément de Git, qui est le plus répandu aujourd'hui.

Système de gestion de version

 Un système de gestion de version est un logiciel permettant aux développeurs de travailler ensemble sur un code source commun, tout en gardant un historique de leur travail, et ce tout au long du développement.
  • Les développeurs peuvent travailler simultanément (même sur des fichiers communs) sans qu'ils voient leurs modifications écrasées.
  • Il est possible de travailler simultanément sur des versions différentes du code.
  • Il est possible de revenir à des versions antérieures du code à tout moment
  • Il est possible de savoir qui a effectué telle ou telle modification, quand, où est pourquoi.
 Il existe deux types de systèmes de gestion de version, le système centralisé ou décentralisé.

Système centralisé vs Système décentralisé

 Le système centralisé (CVS, SVN) oblige à avoir un serveur principal tout le temps accessible tandis que le système décentralisé (Git, Mercurial...) permet de travailler sans, même si un serveur central permet tout de même la mise en commun du code.
 De nos jours, ce sont les systèmes décentralisés qui sont le plus souvent utilisés (du moins pour les nouveaux projets) car plus souples et avec des fonctionnalités plus poussées.

Configuration

 Commençons par quelques opérations de configuration avec l'outil en ligne de commande. Ces commandes sont exécutables dans un terminal classique sur Linux et Mac. Sur Windows, on utilisera Git Bash ou Git Shell.
  • git config --global user.name "Votre nom"
  • git config --global user.email "Votre email"
  • git config --global color.ui auto
 Les deux premières commandes permettront de mieux vous identifier lors de vos différentes actions tandis que la dernière permet d'avoir un affichage coloré au niveau de la ligne de commande.

Les Repository

 Un Repository ou dépôt rassemble la totalité des fichiers et répertoires associés à un projet, ainsi que l'historique de toutes les modifications effectuées (appellées commits). Les fichiers contenant les données sur l'historique sont situées dans le répertoire .git.
  • Créer un nouveau repository ou dépôt local dans le répertoire courant
  • git init nom_du_repository
  • Télécharger un repository existant
  • git clone url_du_repository

Les conflits

 Tout va bien lorsqu'on travaille sur des fichiers différents, ou sur des parties différentes d'un même fichier. Cependant, il peut y avoir des conflits lorsque plusieurs personnes modifie la même ligne d'un même fichier.
 Git n'arrive pas à fusionner automatiquement les contenus qu'il faut fusionner manuellement...

Gestion des conflits

 Lors d'un conflit, Git vous redonne donc la main afin que vous résolviez le(s) conflit(s) vous-même en fournissant à l'intérieur du fichier incriminé, les modifications qui ont été faites des deux côtés. Cela ressemble à ça :
Fichier noms.txt
DURAND
<<<<<<< HEAD
VILLOUD
=======
FAURE
>>>>>>> 97ef45b...

  •  
  • Où pointe votre copie locale
  • Votre modification
  • Séparateur
  • Modification distante
  • hash du commit distant en cause

Pour résoudre le conflit :

  • Modifier le fichier manuellement en faisant attention à bien garder les modifications distantes et les vôtres
  • Ne pas oublier de supprimer les éléments ajoutés par Git (<<<<<< HEAD...)
  • git add noms.txt
  • git commit
  • git push
 Il est alors nécessaire de résoudre manuellement le conflit lorsque Git n'a pas réussi à le faire automatiquement. Il faut être particulièrement attentif lors de la résolution de conflit afin de prendre en compte toutes les modifications et ne pas supprimer celles des autres personnes.

Gestion des conflits (2)

 Cela peut vite devenir compliqué et illisible lorsqu'on doit résoudre plusieurs conflits. On peut donc utiliser un logiciel tierce pour nous aider à le faire. La plupart des IDE ou des GUI pour Git possèdent leur outil.
 Vous pouvez aussi paramétrer Git pour lancer un logiciel de résolution de conflits (comme Meld, voir ici pour un exemple de configuration) lorsqu'il détecte un conflit.

Les branches

 Afin notamment d'éviter de systématiquement devoir récupérer les changements des autres (et régler les potentiels conflits) pour pousser les siens, on peut utiliser le mécanisme des branches. Cela permet aussi de pouvoir pousser son code même s'il n'est pas terminé sans pour autant impacter les autres développeurs.
 A la création d'un repository, une seule branche existe, la branche master. Ce mécanisme de branches permet également de gérer le développement parallèle de plusieurs fonctionnalités, ou la maintenance des différentes versions d'un logiciel. Pour approfondir voir les différents workflows.

Revenir en arrière

 Il existe plusieurs cas pour lesquels un développeur souhaite revenir en arrière. Voyons les commandes en fonction des cas.
  • Consulter une autre version du code
  • Mettre de côté les modifs non commitées pour les appliquer plus tard
  • Annuler tous les commits suivants (garde les modifications localement)
  • Supprime tous les commits suivants (donc toutes les modifications)
  • Supprimer/Déplacer un fichier
  • Supprimer un fichier uniquement de git
  • Inverser un commit
  • git checkout [hash-commit|nom-branche|nom-tag]
  • git stash => git stash pop
  • git reset [hash-commit]
  • git reset --hard [hash-commit]
  • git rm fichier / git mv fichier fichier2
  • git rm --cached fichier
  • git revert mon-commit
 Quand vous utiliser un IDE prenant en charge Git, certaines de ces opérations sont faites toutes seules, lorsque vous supprimez ou déplacez un fichier.
